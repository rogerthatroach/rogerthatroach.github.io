## 1. Introduction

Text-to-SQL — the task of translating natural language questions into executable
SQL queries — has seen significant progress through large language models. DIN-SQL
<CitationRef id={1} />, DAIL-SQL <CitationRef id={2} />, and C3 <CitationRef id={3} /> achieve execution accuracies above 85% on the Spider
benchmark <CitationRef id={4} />. However, production deployment in regulated financial environments
exposes failure modes that benchmark evaluations do not capture:

1. **Ambiguous KPI references** that map to multiple valid metrics
1. **Evolving taxonomies** where new KPIs are continuously added to the catalog
1. **Injection vulnerabilities** from unguarded SQL generation
1. **Non-deterministic outputs** that cannot be regression-tested
1. **Displacive errors** — a confidently wrong answer is worse than no answer

We address these failures through decomposition: rather than treating text-to-SQL
as a single function <InlineEquation html={tex("f: \\text{NL} \\rightarrow \\text{SQL}")} />, we factor it into
five stages, each with explicit contracts, testable invariants, and formal safety
properties.

## 2. Problem Formulation

### 2.1 The Benchmarking Domain

The system operates on a financial benchmarking database containing Key Performance
Indicators (KPIs) derived from regulatory filings of major financial institutions.
Let:

- <InlineEquation html={tex("\\mathcal{K} = \\{k_1, k_2, \\ldots, k_n\\}")} /> be the **KPI catalog**, where each <InlineEquation html={tex("k_i")} /> has a name <InlineEquation html={tex("n_i")} />, definition <InlineEquation html={tex("d_i")} />, and metadata <InlineEquation html={tex("m_i")} /> (reporting period, entity, unit)
- <InlineEquation html={tex("\\mathcal{V}")} /> be the **value store**, containing time-indexed KPI values for each institution
- <InlineEquation html={tex("\\mathcal{Q}")} /> be the space of natural language queries

The system must implement <InlineEquation html={tex("f: \\mathcal{Q} \\rightarrow \\mathcal{R}")} /> where <InlineEquation html={tex("\\mathcal{R}")} /> is the space of formatted benchmark
comparisons.

### 2.2 Formal Requirements

<TheoremBlock variant="definition" number={1} title="Injection Safety">
  <div>
    No user input string <InlineEquation html={tex("q \\in \\mathcal{Q}")} /> can cause
    execution of SQL operations outside the predefined operation set <InlineEquation html={tex("\\mathcal{O} = \\{\\text{SELECT}\\}")} />:
  </div>
  <MathBlock html={tex("\\forall q \\in \\mathcal{Q}: \\text{op}(\\text{SQL}(q)) \\in \\mathcal{O}", true)} />
</TheoremBlock>

<TheoremBlock variant="definition" number={2} title="Schema Compliance">
  <div>
    Generated SQL may reference only tables <InlineEquation html={tex("T_{\\text{allowed}}")} /> and columns <InlineEquation html={tex("C_{\\text{allowed}}")} /> in the registered schema:
  </div>
  <MathBlock html={tex("\\forall q \\in \\mathcal{Q}: \\text{tables}(\\text{SQL}(q)) \\subseteq T_{\\text{allowed}} \\wedge \\text{cols}(\\text{SQL}(q)) \\subseteq C_{\\text{allowed}}", true)} />
</TheoremBlock>

<TheoremBlock variant="definition" number={3} title="Disambiguation Correctness">
  <div>
    If the system resolves an ambiguous KPI reference, the resolved KPI must be the
    one the user intended with probability at least <InlineEquation html={tex("1 - \\epsilon")} /> for a configurable confidence threshold <InlineEquation html={tex("\\epsilon")} />:
  </div>
  <MathBlock html={tex("P(\\hat{k} = k^*) \\geq 1 - \\epsilon", true)} />
  <div>
    Otherwise, the system must request clarification rather than guess.
  </div>
</TheoremBlock>

<TheoremBlock variant="definition" number={4} title="Testability">
  <div>
    Every pipeline stage must be independently testable with deterministic inputs and
    outputs:
  </div>
  <MathBlock html={tex("\\forall \\text{ stage } s_i: \\exists \\text{ test suite } T_i \\text{ s.t. } s_i \\text{ is deterministic on } T_i", true)} />
</TheoremBlock>

## 4. Stage 1: Intent Parsing

### 4.1 Structured Extraction

The intent parser maps a natural language query to a structured intent tuple:

<MathBlock html={tex("s_1: \\mathcal{Q} \\rightarrow \\mathcal{I} = (\\mu, \\tau, \\delta, \\omega)", true)} />

where:

- <InlineEquation html={tex("\\mu \\in \\mathcal{M}")} /> is the **metric reference** (the KPI the user is asking about, in their words)
- <InlineEquation html={tex("\\tau \\in \\mathcal{T}")} /> is the **temporal scope** (quarter, year, range, trend)
- <InlineEquation html={tex("\\delta \\in \\Delta")} /> is the **comparison dimension** (peer vs. self, absolute vs. relative)
- <InlineEquation html={tex("\\omega \\in \\Omega")} /> is the **output type** (table, chart, single value, narrative)

### 4.2 Constrained Generation

The LLM is prompted with a schema definition of <InlineEquation html={tex("\\mathcal{I}")} /> and instructed to output a JSON object
conforming to the schema. This is constrained generation — the model's output
space is restricted to valid intent tuples.

**Validation.** The JSON output is validated against the schema before
proceeding. If validation fails (missing fields, invalid enum values), the system
retries with the validation error appended to the prompt. After <InlineEquation html={tex("k_{\\max}")} /> retries, it requests clarification from the
user.

**Ambiguity Detection.** If the metric reference <InlineEquation html={tex("\\mu")} /> is empty or too vague (e.g.,
"efficiency" with no qualifying context), the system returns a
clarification request rather than proceeding with a guess. This is Stage 1's
contribution to Requirement 3.

## 5. Stage 2: KPI Detection via Embedding Similarity

### 5.1 Embedding Space

Each KPI <InlineEquation html={tex("k_i \\in \\mathcal{K}")} /> is represented by an
embedding vector computed from its name, definition, and metadata:

<MathBlock html={tex("\\mathbf{e}_i = \\text{embed}(n_i \\oplus d_i \\oplus m_i) \\in \\mathbb{R}^d", true)} />

where <InlineEquation html={tex("\\oplus")} /> denotes string concatenation with separator
tokens and <InlineEquation html={tex("\\text{embed}: \\text{String} \\rightarrow \\mathbb{R}^d")} /> is a pre-trained sentence embedding model.

The user's metric reference <InlineEquation html={tex("\\mu")} /> is similarly
embedded:

<MathBlock html={tex("\\mathbf{e}_\\mu = \\text{embed}(\\mu)", true)} />

### 5.2 Similarity Search

The candidate set is retrieved by cosine similarity:

<MathBlock html={tex("\\text{sim}(k_i, \\mu) = \\frac{\\mathbf{e}_i \\cdot \\mathbf{e}_\\mu}{\\|\\mathbf{e}_i\\| \\|\\mathbf{e}_\\mu\\|}", true)} />

<MathBlock html={tex("\\mathcal{K}_c = \\{k_i \\in \\mathcal{K} \\mid \\text{sim}(k_i, \\mu) \\geq \\tau_{\\min}\\}", true)} />

where <InlineEquation html={tex("\\tau_{\\min}")} /> is the minimum similarity threshold.

### 5.3 Why Embeddings Over Keyword Matching

<TheoremBlock variant="proposition" number={3} title="Embedding Robustness to Synonymy">
  <div>
    For semantically equivalent metric references <InlineEquation html={tex("\\mu_1 \\equiv_{\\text{sem}} \\mu_2")} /> (e.g.,
    "NIM" and "Net Interest Margin"), embedding similarity
    preserves retrieval:
  </div>
  <MathBlock html={tex("\\mu_1 \\equiv_{\\text{sem}} \\mu_2 \\implies |\\text{sim}(k, \\mu_1) - \\text{sim}(k, \\mu_2)| < \\delta", true)} />
  <div>
    for small <InlineEquation html={tex("\\delta")} />, assuming the embedding model captures
    semantic similarity (a property validated empirically for modern sentence
    transformers <CitationRef id={5} />).
  </div>
</TheoremBlock>

Keyword matching fails here because "NIM" shares zero tokens with
"Net Interest Margin." Edit distance fails because the strings are not
typos of each other. Embeddings succeed because they encode semantic meaning, not
surface form.

### 5.4 The Scaling Problem

As the KPI catalog <InlineEquation html={tex("|\\mathcal{K}|")} /> grows, the probability
of near-duplicate entries increases. Consider:

- "Net Interest Margin"
- "Net Interest Margin (Adjusted)"
- "Net Interest Margin (Domestic Only)"
- "NIM — Regulatory Basis"

These are distinct KPIs with different definitions and values, but their embeddings
cluster tightly in <InlineEquation html={tex("\\mathbb{R}^d")} />. When <InlineEquation html={tex("|\\mathcal{K}_c| > 1")} /> and the top candidates are close in
similarity score, we enter the disambiguation stage.

## 6. Stage 3: LLM-Assisted Disambiguation

### 6.1 Triggering Condition

Disambiguation is invoked when the candidate set is ambiguous:

<MathBlock
  html={tex("\\text{disambiguate?} = \\begin{cases} \\text{no} & \\text{if } |\\mathcal{K}_c| = 1 \\\\ \\text{no} & \\text{if } |\\mathcal{K}_c| > 1 \\wedge \\text{sim}(k_1, \\mu) - \\text{sim}(k_2, \\mu) > \\gamma \\\\ \\text{yes} & \\text{otherwise} \\end{cases}", true)}
/>

where <InlineEquation html={tex("k_1, k_2")} /> are the top-2 candidates sorted by
similarity, and <InlineEquation html={tex("\\gamma")} /> is the **confidence gap threshold**. If the top candidate is sufficiently
dominant, we accept it without LLM disambiguation.

### 6.2 The Disambiguation Protocol

When triggered, the LLM receives:

- The user's original query <InlineEquation html={tex("q")} />
- The candidate KPI names <InlineEquation html={tex("\\{n_i : k_i \\in \\mathcal{K}_c\\}")} />
- The candidate KPI definitions <InlineEquation html={tex("\\{d_i : k_i \\in \\mathcal{K}_c\\}")} />

Critically, the LLM does **not** receive KPI values or any data from <InlineEquation html={tex("\\mathcal{V}")} />. This preserves the data confidentiality
property (cf. companion paper on LLM-as-Router architectures).

The LLM outputs a selection <InlineEquation html={tex("\\hat{k} \\in \\mathcal{K}_c")} /> and a confidence score <InlineEquation html={tex("p \\in [0, 1]")} />.

### 6.3 Confidence Thresholding

<TheoremBlock variant="definition" number={6} title="Disambiguation Decision Function">
  <div>The disambiguation outcome is:</div>
  <MathBlock
    html={tex("\\text{outcome}(\\hat{k}, p) = \\begin{cases} \\text{accept}(\\hat{k}) & \\text{if } p \\geq 1 - \\epsilon \\\\ \\text{clarify}(\\mathcal{K}_c) & \\text{if } p < 1 - \\epsilon \\end{cases}", true)}
  />
  <div>
    where <InlineEquation html={tex("\\epsilon")} /> is the maximum acceptable error
    probability from Requirement 3.
  </div>
</TheoremBlock>

<TheoremBlock variant="proposition" number={4} title="Disambiguation Safety">
  <div>
    If the LLM's confidence score is calibrated (i.e., <InlineEquation html={tex("P(\\hat{k} = k^* \\mid p) \\geq p")} /> for all confidence
    levels <InlineEquation html={tex("p")} />), then the disambiguation decision function satisfies Requirement 3:
  </div>
  <MathBlock html={tex("P(\\hat{k} = k^*) \\geq 1 - \\epsilon \\text{ whenever outcome} = \\text{accept}", true)} />
</TheoremBlock>

<TheoremBlock variant="proof">
  <div>
    The accept condition requires <InlineEquation html={tex("p \\geq 1 - \\epsilon")} />. By
    calibration, <InlineEquation html={tex("P(\\hat{k} = k^* \\mid p) \\geq p \\geq 1 - \\epsilon")} />. <InlineEquation html={tex("\\square")} />
  </div>
</TheoremBlock>

In practice, LLM confidence scores are not perfectly calibrated <CitationRef id={6} />, so we set <InlineEquation html={tex("\\epsilon")} /> conservatively (we use <InlineEquation html={tex("\\epsilon = 0.05")} />) and validate calibration empirically on
a held-out query set.

### 6.4 Model Readiness as a Precondition

The disambiguation safety property in Proposition 4 is *conditional on
calibration* — and calibration is an empirical property of the underlying LLM,
not of the architecture. The same five-stage pipeline executed against a less-
calibrated model fails Requirement 3 even though it satisfies the architecture
specification.

This is not theoretical. An earlier prototype of this system, executed against
GPT-4o, exhibited two pathological failure modes on the disambiguation step:

1. **Over-confident misselection.** Near-duplicate KPI candidates (e.g.
   *Net Interest Margin* vs. *Net Interest Margin — Adjusted*) elicited high
   confidence scores even when the wrong candidate was selected.
   <InlineEquation html={tex("P(\\hat{k} = k^* \\mid p)")} /> diverged from <InlineEquation html={tex("p")} /> on the catalog clusters that mattered most.
1. **Pathological caution.** Across other clusters, the model returned
   uniformly low confidence for unambiguous candidates, triggering clarification
   requests where none should have been needed.

Either failure violates the disambiguation safety property. The architecture
was structurally correct; the model was not yet a fit. We deferred production
deployment of v2 until a model with sufficient calibration on this catalog
domain became available.

GPT-4.1 closed both gaps to within tolerance on a held-out evaluation set of
~200 disambiguation queries (calibration ECE below 0.05; over-confidence on
near-duplicate clusters within margin of <InlineEquation html={tex("\\epsilon")} />).

The general lesson is that an architecture with formal safety properties still
inherits one empirical dependency: the calibration of any LLM used in a
disambiguation-or-clarify decision. Architecture specifies the *form* of safety;
the model has to clear the *bar* the form requires.

## 7. Stage 4: Guardrailed SQL Generation

### 7.1 Template-Based Generation

Once the KPI is resolved, SQL generation uses a hybrid approach: a template library
provides the structural skeleton, and parameterization fills in the specific values.

<TheoremBlock variant="definition" number={7} title="SQL Template">
  <div>
    A template <InlineEquation html={tex("T")} /> is a SQL string with typed placeholders:
  </div>
  <MathBlock html={tex("T(\\text{kpi\\_id}: \\text{int},\\; \\text{period}: \\text{date},\\; \\text{entity}: \\text{string}) = \\text{SELECT } \\ldots \\text{ WHERE kpi\\_id = :kpi\\_id AND } \\ldots", true)} />
  <div>
    Templates are pre-authored and reviewed for each query pattern (peer comparison,
    time series trend, single-point lookup, cross-entity ranking).
  </div>
</TheoremBlock>

### 7.2 Formal Safety Properties

<TheoremBlock variant="theorem" number={3} title="Injection Impossibility">
  <div>
    If all user-supplied values are passed as parameterized bindings (not
    string-interpolated into the SQL), then SQL injection is impossible:
  </div>
  <MathBlock html={tex("\\forall q \\in \\mathcal{Q}: \\nexists \\text{ substring } s \\text{ of } q \\text{ that executes as SQL}", true)} />
</TheoremBlock>

<TheoremBlock variant="proof">
  <div>
    Parameterized queries separate code (the SQL template) from data (the parameter
    values) at the database driver level <CitationRef id={7} />. The driver processes the SQL template as
    a prepared statement, compiling it into an execution plan before binding parameter
    values. Parameter values are treated as data literals, not as SQL syntax.
    Therefore, no content of <InlineEquation html={tex("q")} /> — regardless of its structure
    or embedded SQL keywords — can alter the query's syntactic structure or
    execution plan.
  </div>
</TheoremBlock>

<TheoremBlock variant="theorem" number={4} title="Schema Compliance">
  <div>
    If the template library <InlineEquation html={tex("\\mathcal{T}")} /> references only
    tables in <InlineEquation html={tex("T_{\\text{allowed}}")} /> and columns in <InlineEquation html={tex("C_{\\text{allowed}}")} />, and templates are selected (not
    generated), then Requirement 2 is satisfied:
  </div>
  <MathBlock html={tex("\\forall T \\in \\mathcal{T}: \\text{tables}(T) \\subseteq T_{\\text{allowed}} \\wedge \\text{cols}(T) \\subseteq C_{\\text{allowed}}", true)} />
  <MathBlock html={tex("\\text{template\\_select}: \\mathcal{I} \\rightarrow \\mathcal{T} \\implies \\text{tables}(\\text{SQL}(q)) \\subseteq T_{\\text{allowed}}", true)} />
</TheoremBlock>

<TheoremBlock variant="proof">
  <div>
    The SQL execution uses only templates from <InlineEquation html={tex("\\mathcal{T}")} />. Each template is authored and reviewed to
    reference only allowed tables and columns. Since no dynamic table or column names
    enter through parameters (which are values, not identifiers), the schema boundary
    is preserved.
  </div>
</TheoremBlock>

### 7.3 Validation Layer

Even with template-based generation, a validation layer provides defense-in-depth:

1. **Whitelist check (allow-list):** Parse the generated SQL AST and verify all table and column references against <InlineEquation html={tex("T_{\\text{allowed}}")} /> and <InlineEquation html={tex("C_{\\text{allowed}}")} />
1. **Operation check:** Verify the top-level operation is SELECT (no INSERT, UPDATE, DELETE, DROP)
1. **Structure check:** Compare the generated SQL structure against the expected template structure using tree edit distance on the AST
1. **Parameter type check:** Verify each bound parameter matches the expected type from the template definition
1. **Forbidden-keyword filter (deny-list):** Reject SQL containing operations or schema mutations the allow-list might miss (e.g. `CREATE`, `GRANT`, `EXEC`). Defense-in-depth — redundant with the whitelist by design

If any check fails, the query is rejected and logged for review — it never reaches
the database. The validation layer is intentionally redundant: an allow-list defines what is permitted; a deny-list catches what slipped past the allow-list. Either alone is sufficient in theory; both together is what gets shipped to production.

## 8. Stage 5: Deterministic Result Formatting

The final stage transforms raw query results into the output format <InlineEquation html={tex("\\omega")} /> specified in the intent. This stage is entirely
deterministic — no LLM involvement:

<MathBlock html={tex("s_5: (\\text{QueryResult}, \\omega) \\rightarrow \\mathcal{R}", true)} />

Output types include comparison tables, trend charts, summary statistics, and
narrative descriptions (generated from templates, not LLMs).

## 9. On the Two-Week Delivery

The second version of this system was designed and deployed in two weeks. This
timeline merits explanation because it appears to contradict the depth of
engineering described above.

Four factors made this possible:

**Factor 1: Domain maturity.** Version 1 had been in production for
several months. The KPI catalog, data layer, entitlement model, and deployment
infrastructure were stable. The rewrite replaced the query interface, not the
foundation.

**Factor 2: Pre-crystallized architecture.** The five-stage
decomposition, the embedding approach, the confidence thresholding protocol, and the
guardrail layers had been explored and decided during months of research preceding
the build. The architecture was ready; it needed to be written, not discovered.

**Factor 3: Model readiness.** An earlier prototype with GPT-4o had
demonstrated the architecture but failed the calibration bar Requirement 3
demands (see §6.4). When GPT-4.1 cleared that bar on a held-out evaluation,
the shelved design became viable as-is. The cost of having prototyped early
was almost zero; the value when calibration crossed threshold was several
months of ramp-up that did not have to be repeated.

**Factor 4: Framework standardization.** The codebase was built within
an internal platform that handled authentication, entitlements, and deployment.
Infrastructure was not in scope.

The lesson: speed in delivery comes from depth in preparation. The two weeks of
building was enabled by months of thinking.

## 10. Related Work

**Text-to-SQL benchmarks.** Spider <CitationRef id={4} /> and WikiSQL <CitationRef id={8} /> evaluate
end-to-end accuracy but do not measure safety properties (injection resistance,
schema compliance) or disambiguation quality. Our pipeline is designed for these
properties first, accuracy second.

**Decomposed approaches.** DIN-SQL <CitationRef id={1} /> decomposes text-to-SQL into
sub-problems but uses LLM generation at each stage. Our approach restricts LLM use
to intent parsing and disambiguation, using deterministic templates for SQL
generation.

**Embedding-based retrieval.** Dense retrieval methods <CitationRef id={9} /> using
sentence transformers have shown strong performance for semantic matching tasks. We
apply this specifically to KPI catalog matching with a disambiguation overlay.

**SQL safety.** OWASP SQL injection prevention <CitationRef id={7} /> establishes
parameterization as the primary defense. Our contribution is integrating this with
LLM-generated queries where the attack surface is the natural language input, not a
traditional form field.

## 11. Conclusion

The five-stage decomposition transforms text-to-SQL from a monolithic generation
task into a pipeline with formal safety properties. By factoring the problem into
intent parsing, embedding-based detection, confidence-thresholded disambiguation,
template-based generation with parameterization, and deterministic formatting, we
achieve injection impossibility, schema compliance, and disambiguation correctness —
properties that are provable, not aspirational.

The key insight: **decomposition is itself the primary guardrail.** Each stage has a narrow contract, a testable interface, and clear failure modes.
When something goes wrong, you know which stage failed. When the catalog grows, you
know which stage to extend. When requirements change, you know which stage to modify
without risking the others.
