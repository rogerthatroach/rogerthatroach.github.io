In November 2025, I wrote a one-page plan for an AI system that would transform how RBC drafts Project Approval Requests. Five months later, it shipped bank-wide.

This is the story of how it happened, what I learned about scaling an intern&rsquo;s proof-of-concept into an enterprise product, and why the hardest part had nothing to do with technology.

## The problem

Project Approval Requests (PARs) are how major banks govern significant initiatives. They&rsquo;re high-stakes documents that require institutional knowledge, policy compliance, and cross-referencing dozens of templates, rules, and historical examples. The drafting process took weeks. Most of that time was spent hunting for the right template, resolving contradictions between policy documents, and guessing which fields applied to which project type.

This wasn&rsquo;t a search problem. It was an orchestration problem.

## The spark

During the 2025 summer intern program, one of the four interns I was managing prototyped a concept: what if an AI assistant could guide you through a PAR step by step? The prototype was rough, a basic chatbot with a single RAG layer, but the insight was sharp. The user didn&rsquo;t need answers. They needed guidance through a process.

I saw it could scale from a one-team prototype to a bank-wide platform. Not a chatbot. An agentic platform that could understand where you were in the workflow, pull the right context at the right moment, resolve conflicts between policies automatically, and adapt to any PAR type across any business line.

## The architecture decisions

**The governance constraint that shaped everything.** PAR Drafting Assistant was the first true agentic system approved for production at the bank. The governance envelope for v1 was a single agent, one approved scope — no multi-agent orchestration. Every architectural decision below had to deliver agentic *behaviour* (branching dialog, parallel specialized work, multi-turn state) without crossing into multi-agent *shape*. That constraint is the spine of the system.

**Why LangGraph over CrewAI / AutoGen.** Two reasons, in that order. First: at evaluation time LangGraph was the most mature, stable graph-orchestration library — production-deployed elsewhere, active maintenance. In a regulated-finance first-deployment context, stable-and-maintained outranks elegant-on-paper. Second: the drafting session is shaped like a graph, not a pipeline — template selection, field-group retrieval, extraction, sometimes loop back to clarify open follow-ups. LangGraph&rsquo;s checkpointer wires directly into Postgres, so session durability comes for free. CrewAI and AutoGen were considered but not deeply evaluated — multi-agent primitives wouldn&rsquo;t have cleared the single-agent governance bar anyway.

**Why MCP tools as the action boundary.** Every action the agent takes is a typed, logged tool invocation — template selection, retrieval, compression, extraction, merge, coverage. No invisible prompt engineering. With MCP, the property *every action lands in the audit log* is structural, not aspirational — the engine dispatches actions only through the tool registry, so no code path exists outside it. In a regulated environment, that guarantee isn&rsquo;t optional.

**Why field-group retrieval beat flat RAG.** A PAR has hundreds of target fields across templates, most of them partially repeating across template types. Flat top-k retrieval surfaces chunks from unrelated fields — too much relevant-looking text, not enough that maps to a specific field. We organized the knowledge base around the *target* the LLM extracts toward: field groups aggregate logically related fields across templates. Stage-1 retrieval picks relevant groups for the session; stage-2 retrieves top-10 chunks per group; custom compression fits them into a Sonnet-4.5 system prompt that carries up to 20 fields of rich metadata. Flat RAG would work for a Q&A chatbot. It doesn&rsquo;t work for structured extraction across a template with categorically different fields.

**How we got N parallel extraction inside a single-agent envelope.** The problem shape wanted specialist sub-agents per field domain. Governance said no. The architecture that slipped through: the relevant group count N from stage-1 retrieval becomes the parallelism — up to N Sonnet-4.5 calls fire at once, each carrying its group&rsquo;s compressed chunks, ≤20 fields of metadata, and guardrails + few-shot good/bad examples scoped to its group. Each returns a JSON dictionary. The merge is a dict union (groups disjoint by construction). From the outside it looks agentic: branching dialog, parallel specialized work. From governance review, it&rsquo;s one agent calling tools.

**Why PostgreSQL is the backbone.** One store for LangGraph checkpoints, tool-invocation logs, raw ingested content, mapped intermediates, embeddings (via pgvector), and audit trail. Unusual — retrieval usually sits in a dedicated vector store — but at PAR scale pgvector is sufficient, and co-locating state with embeddings with audit means a full session provenance is a handful of joins. Backup / replication / access control only solved once.

**Context isolation.** The LLM operates at intent parsing and extraction nodes only. Sensitive data access happens exclusively through MCP tools in a separate context. This isn&rsquo;t a prompt-engineering trick; it&rsquo;s a structural invariant the graph topology enforces. The formal post proves it by construction. The informal version: you can&rsquo;t leak what you can&rsquo;t reach.

## The leadership lessons

The hardest part wasn&rsquo;t the architecture. It was three things.

**Scoping.** The intern prototype tried to do everything. The production version does one thing exceptionally well: guide a user through a PAR draft, step by step, with the right context at the right moment. Saying no to features is harder than building them. Every &ldquo;what if we also&hellip;&rdquo; is a seductive detour. I learned to ask: does this serve the guided-drafting promise, or dilute it? The answers were usually no.

**Parallel execution.** I was running PAR Drafting Assistant alongside CFO Analytics Engine (the production analytics platform) and the tail end of Benchmarking Engine v2. The only way it worked was because the architecture was clean enough that the work could be parallelized. Clean separation of concerns isn&rsquo;t just good engineering. It&rsquo;s a management strategy. When your components have narrow interfaces, multiple workstreams can progress independently without stepping on each other. I didn&rsquo;t appreciate this until I was trying to run three products with one direct report and a rotating cast of interns.

**Trusting the origin.** An intern had the initial idea. Some leaders would have absorbed it, renamed it, and made it &ldquo;theirs.&rdquo; I think that&rsquo;s how you kill innovation. The best thing I did was resource the idea, scale the vision, and make sure everyone, including senior leadership, knew where it started. That&rsquo;s not generosity. It&rsquo;s how you build a team where people bring their best ideas to work. If the intern who sparks your next enterprise product gets a promotion, a reference, and visible credit, the next intern does too. And the one after that.

## The result

PAR Drafting Assistant&rsquo;s pilot launched in April 2026, less than three months from when real development started after holidays and team restructuring. Enterprise rollout across RBC business lines continues through Q2 and Q3 2026. What was weeks of manual drafting becomes a guided interactive session — one that cites every extracted field back to a policy, guideline, or few-shot example. It targets key PAR authors: project sponsors, senior leaders, PMO teams, across all RBC business lines. And crucially, v1 is the substrate the v2 multi-agent &ldquo;skills&rdquo; framework is being built on — not a throwaway, but the part of v2 that already passed governance review.

## What I&rsquo;d do differently

Start the formal architecture document earlier. We had the vision in November 2025 but didn&rsquo;t start real development until February 2026 because of holidays and team restructuring. Having a detailed technical design ready to go the moment capacity freed up would have shaved two to three weeks off the timeline.

Involve end users (PAR authors) earlier in the design process. We built based on our understanding of the pain points, which was accurate. But earlier user feedback would have caught edge cases we found later.

## The pattern

PAR Drafting Assistant follows the same closed-loop pattern as every system I&rsquo;ve built.

- **Sense.** Ingest uploaded material (pptx / docx / pdf / txt / images via GPT-4-turbo OCR); template MCP tool classifies the PAR type via decision-tree dialog.
- **Model.** Stage-1 retrieval picks relevant field groups; stage-2 retrieves top-10 chunks per group; custom compression fits them into scoped prompts.
- **Optimize.** N parallel Sonnet-4.5 extraction calls; dict-union merge; coverage analyzer surfaces intelligent follow-ups.
- **Act.** Rendered PAR draft with every field cited back to its source, auditable end to end via one Postgres store.

The technology is different from Particle Swarm Optimization on a 900MW coal plant. The architecture is the same. The formal post [makes that isomorphism rigorous](/blog/closed-loop). This post is the version I&rsquo;d tell you at lunch.
