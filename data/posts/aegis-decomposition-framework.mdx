Finance analysts ask a question with subtle ambiguity baked in: *"how does our cost-to-income ratio compare against the peer average this quarter?"* &mdash; a sentence that maps cleanly to one KPI in some banks and to three near-duplicate KPIs in others, depending on how each bank reports its Supplementary Financial Package. The answer has to be a single number, on a deterministic computation path, with the resolution from natural language to specific KPI defensible under audit.

The industry pattern here is straightforward: prompt an LLM with a schema definition and the user&rsquo;s question; let the model emit SQL; run it. Fast to prototype. Easy to demo. Impossible to ship inside regulated finance &mdash; chain-of-thought is not an audit trail; LLM-emitted SQL is not a safety property; *"the model usually gets it right"* is not a confidence interval.

Aegis went the other way. Five sequential stages, two dashed walls, LLM use bounded to two stages (and the second only when a confidence gate fires). Templates produce SQL; parameters bind values; deterministic code formats the result. This post unpacks the four architectural decisions that separate decomposition-as-guardrail from the seductive single-prompt pattern &mdash; with the alternatives that were seriously considered, the constraints each one answered, and the reasoning that produced the call. The narrative of *when* those calls landed &mdash; and the role GPT-4.1&rsquo;s calibration played in unlocking a shelved design &mdash; lives in the [builder companion](/blog/aegis-v2-velocity). The formal proofs of injection impossibility, schema compliance, and bounded disambiguation live in the [formal post](/blog/text-to-sql).

Four constraints shaped every decision below. Two regulatory, one hard-technical, one soft-organizational. Every downstream choice satisfies these first; elegance second.

## The four architectural decisions

Four calls separate decomposition-as-guardrail from the single-prompt pattern. Each one mapped to an explicit options comparison before it landed.

### Decision 1 · Orchestration shape

Does the system give the LLM the natural-language question and ask for SQL? Or does it decompose the work into stages and bound the LLM&rsquo;s role?

<DecisionRationale option="Decomposed five-stage pipeline (LLM use bounded to S1 + S3)">
Three reasons, in order. First &mdash; **decomposition is the guardrail.** Each stage has a narrow contract, a testable interface, and clear failure modes. When the disambiguation gate fires, the SQL stage has not run yet. When SQL gen fires, the KPI is already resolved and validated. Errors do not propagate through the system; they are caught at the stage where they originate. The architecture is auditable because the data flow is auditable.

Second &mdash; **the safety properties prove per stage.** Injection impossibility comes from the SQL stage&rsquo;s template + parameter contract; it does not depend on the LLM behaving well. Schema compliance comes from the template library + AST validator; same. Disambiguation safety comes from the confidence gate + ε threshold; same. Each property is local to a stage. The proofs are short. The audit is short.

Third &mdash; **catalog growth is data, not code.** Adding a new KPI from a new SFP is a row in the catalog table plus an embedding. No new rule, no new template (templates are organized by query *shape*, not by KPI). The system absorbs catalog growth without architectural amendment &mdash; which is exactly what v1 could not do.

The cost is the upfront template library and the per-stage contract design. Both are one-time engineering investments. They paid back in the first month of operation.
</DecisionRationale>

### Decision 2 · KPI detection method

Once the intent is parsed, the system has a metric reference (μ &mdash; the user&rsquo;s phrase for the KPI they want). The catalog has thousands of entries with overlapping names. How does the system find the right one?

  decision="How does the system map a metric reference to a candidate set of KPIs in the catalog?"
  options={[
    {

      summary: 'Skip embeddings; pass the full catalog into the LLM context and have it return the matched KPI.',
      pros: [
        'No embedding infrastructure',
      ],
      cons: [
        'Catalog size grows; context window cost grows linearly',

        'No confidence score on the *detection* step; only on the final answer',
      ],
    },
  ]}
/>

<DecisionRationale option="Sentence embeddings + cosine similarity threshold">
The catalog has the structural feature that breaks keyword matching and saves embedding similarity: synonymy across human conventions. *NIM* and *Net Interest Margin* and *Net Interest Income / Total Assets* are the same concept under three notations; embeddings put them in the same neighborhood, keywords do not.

The detection stage&rsquo;s job is *recall*, not precision. It returns a candidate set <em>K<sub>c</sub></em>, not a single answer. Precision is the next stage&rsquo;s job &mdash; the disambiguation gate decides whether the candidate set is unambiguous (top candidate dominates) or whether the LLM (or, ultimately, the user) needs to break the tie. Separating recall from precision is what lets the embedding stage stay deterministic and lets the disambiguation stage stay LLM-bounded.
</DecisionRationale>

### Decision 3 · SQL generation safety model

The pipeline now has a resolved KPI plus parameters (time window, entity, comparison shape). How does the system produce the actual SQL?

<OptionsConsidered
  decision="How is the SQL string produced, and what makes it safe?"
  options={[
    {
      name: 'Template selection + parameterized binding',
      summary: 'A pre-authored, schema-whitelisted SQL template library covers the query shapes (peer comparison, time series, single-point lookup, ranking). Parameters bind through the DB driver as prepared-statement values; templates are never modified at runtime.',
      chosen: true,
      pros: [
        'SQL injection is impossible by construction — parameter values are never interpreted as SQL syntax (proven in formal post, Theorem 3)',
        'Schema compliance is a static property of the template library — verifiable once, holds always',
        'Templates are reviewable artefacts under regulatory inspection; LLM-generated SQL is not',
        'Defense-in-depth validation layer (whitelist + AST + SELECT-only + parameter type + deny-list) catches anything anomalous before execution',
      ],
      cons: [
        'Template library has to be authored upfront; covers only pre-anticipated query shapes',
        'New query shapes require a new template + review; not auto-emergent',
      ],
    },
    {
      name: 'LLM-generated SQL with post-hoc validation',
      summary: 'Let the LLM emit SQL; sanitize / validate before executing. Reject if validation fails.',
      pros: [
        'Flexible: no upfront template authoring',
        'Familiar from research benchmarks (Spider, WikiSQL evaluations)',
      ],
      cons: [
        'Injection prevention is reactive (catch bad SQL) rather than constructive (cannot generate bad SQL)',
        'Sanitization is a denylist exercise — every new attack vector requires a rule update',

        'Schema compliance is best-effort and probabilistic',
      ],
    },

      name: 'AST-based SQL synthesis from intent',
      summary: 'Build an AST from the parsed intent + resolved KPI; serialize to SQL.',
      pros: [
        'Theoretically sound — the AST captures structure exactly',
        'No string concatenation anywhere',
      ],
      cons: [
        'Materially the same as templates with extra steps — query shapes still have to be enumerated',
        'AST builder becomes the new template library, with worse ergonomics for review',
      ],
    },

<DecisionRationale option="Template selection + parameterized binding">
The mental shift required is recognizing that *the template library is the system&rsquo;s SQL surface*, not the LLM. Once that lands, every safety property follows: injection impossibility because parameters bind through the driver; schema compliance because templates reference only allowed tables and columns; auditability because the template library is a review artefact under version control.

The defense-in-depth validation layer is intentionally redundant. A whitelist defines what is permitted; a deny-list catches what slipped past the whitelist; an AST check verifies structure; a SELECT-only operation check rules out DDL or mutation. Either alone would be sufficient in theory. Both together &mdash; plus parameter type checking &mdash; is what gets shipped to production. The redundancy costs almost nothing and earns the right to claim "impossible by construction" rather than "very unlikely in practice."
</DecisionRationale>

### Decision 4 · Disambiguation contract

Some KPI references are unambiguous. Some are not. What does the system do when it isn&rsquo;t sure?

<OptionsConsidered
  decision="When the KPI candidate set is ambiguous, does the system answer anyway, or does it ask?"
  options={[
    {
      name: 'Confidence-thresholded clarify (accept if p ≥ 1−ε; clarify otherwise)',
      summary: 'LLM disambiguator produces a candidate selection plus a calibrated confidence score. If confidence clears the threshold, accept. If not, return a clarification request listing the candidates with their definitions.',
      chosen: true,
      pros: [
        'Disambiguation safety property is testable: P(k̂ = k* | accept) ≥ 1−ε if calibration holds',
        'Calibration can be empirically validated on held-out queries; bad calibration shows up in metrics, not in user trust',
        'Clarification requests improve over time as the catalog stabilizes — the system learns where the edges live',
        '"Confidently wrong" failure mode is structurally precluded',
      ],
      cons: [
        'Requires LLM calibration that holds across the catalog distribution — empirical, not architectural',
        'Clarification asks add a step in the user flow; some users find this friction',
      ],
    },
    {
      name: 'Always answer (highest-similarity wins)',
      summary: 'Pick the top embedding match. Always emit SQL. Surface alternatives in the response if requested.',
      pros: [
        'Frictionless interaction',
        'No LLM disambiguation cost',
      ],
      cons: [
        'Confidently-wrong failure mode is the *default*: top match on near-duplicates can easily be the wrong KPI',
        'CFO-grade analytics cannot tolerate silent misselection — once trust is broken, it does not return',
      ],
    },
    {
      name: 'Always clarify (return candidates; let the user pick)',
      summary: 'Whenever |K_c| > 1, return the candidate list and have the user choose.',
      pros: [
        'Maximally safe — the user is always in the loop',
        'No LLM disambiguation cost',
      ],
      cons: [
        'Heavy interaction friction for cases where the top match dominates by a wide margin',
        'Ignores the LLM&rsquo;s genuine ability to disambiguate when the question + definitions make the answer obvious',
        'Pushes domain-knowledge load onto users who came to the system to escape it',
      ],
    },
  ]}
/>

<DecisionRationale option="Confidence-thresholded clarify (accept if p ≥ 1−ε; clarify otherwise)">
The disambiguation contract is the single property most CFO-Group analysts care about: *will it confidently give me the wrong number?* The architectural answer is no &mdash; structurally. The confidence gate is the keystone that makes the answer no. Below the threshold, the system never emits an answer; above the threshold, calibration plus a conservative ε ensures the error rate is bounded.

The calibration dependency is what gated the v2 launch. An earlier prototype on GPT-4o exhibited two failure modes the gate could not absorb: over-confidence on near-duplicates (high p, wrong selection &mdash; gate accepts a wrong answer) and pathological caution (low p on unambiguous cases &mdash; gate over-clarifies). Either failure violates the safety property. We deferred deployment until a model with sufficient calibration on the catalog domain was available; GPT-4.1 cleared the bar on a held-out evaluation and the architecture went live as designed.
</DecisionRationale>

## What transfers

The *family* this architecture belongs to is the LLM-as-Router pattern (see [Astraeus practitioner post](/blog/astraeus-llm-as-router-framework) for the cross-domain analytics variant). The general shape: bound the LLM to *intent* and *judgment under uncertainty*; deterministic code does compute and produces SQL/results. Walls separate the LLM context from data and from emitted SQL. Decomposition is the guardrail.

What transfers across instances of the family is the *shape* &mdash; the bounded LLM surface area, the confidence gates where uncertainty triggers clarification, the deterministic compute or generation backbone. What does *not* transfer is the specific load-bearing piece: in Astraeus it is event-level ins-outs math across 40K leaf-level events; in Aegis it is the template library plus parameterized binding. Each system is built around the part that has to be most precise. The LLM does what LLMs do well, in the narrowest possible surface area, behind walls that hold by construction.

The hardest engineering is not the LLM integration. It is figuring out which one piece is load-bearing &mdash; and refusing to let the LLM near it.
