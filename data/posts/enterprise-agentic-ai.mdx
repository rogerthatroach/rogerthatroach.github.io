## 1. Introduction

Enterprise organizations run on structured processes. Project approvals, budget requests, compliance reviews — these workflows are high-stakes, document-heavy, and riddled with institutional knowledge that lives in people's heads rather than systems.

Most teams throw a chatbot at this. But the user doesn't need a Q&A interface. They need a *collaborator* — something that understands where they are in a multi-step process, holds context across the entire session, pulls the right reference material at the right moment, and resolves contradictions between policy documents without the user knowing there was a conflict.

That's a fundamentally different problem than retrieval-augmented generation <CitationRef id={7} />. It's an orchestration problem. This paper formalizes the architectural pattern that solves it.

## 2. Problem Formulation

### 2.1 System Requirements

Let <InlineEquation html={tex("\\mathcal{W}")} /> denote the space of enterprise workflows, <InlineEquation html={tex("\\mathcal{U}")} /> the space of user utterances, <InlineEquation html={tex("\\mathcal{D}")} /> the heterogeneous document corpus, and <InlineEquation html={tex("\\mathcal{S}")} /> the state space tracking workflow progress. We seek a system <InlineEquation html={tex("\\mathcal{F}")} /> that guides users through <InlineEquation html={tex("\\mathcal{W}")} /> while satisfying four formal requirements:

<TheoremBlock variant="definition" number={1} title="System Requirements">
**Requirement 1 (Context Isolation).** The LLM component <InlineEquation html={tex("\\mathcal{L}")} /> must never observe sensitive operational data <InlineEquation html={tex("\\mathcal{D}_{\\text{sensitive}}")} />:

<MathBlock html={tex("\\mathcal{E}(\\mathcal{L}) \\cap \\mathcal{D}_{\\text{sensitive}} = \\emptyset", true)} />

**Requirement 2 (Stateful Orchestration).** The system must maintain workflow state <InlineEquation html={tex("s \\in \\mathcal{S}")} /> across an unbounded number of interaction turns:

<MathBlock html={tex("s_{t+1} = \\delta(s_t, u_t, a_t) \\quad \\text{where } \\delta: \\mathcal{S} \\times \\mathcal{U} \\times \\mathcal{A} \\to \\mathcal{S}", true)} />

**Requirement 3 (Heterogeneous Retrieval).** Retrieval must respect document structure, source provenance, and per-step relevance criteria:

<MathBlock html={tex("\\text{retrieve}(q, s_t) = \\bigoplus_{l=1}^{L} R_l(q, s_t, \\mathcal{D}_l)", true)} />

**Requirement 4 (Auditability).** Every system action must be a logged, typed tool invocation <CitationRef id={1} /><CitationRef id={2} />:

<MathBlock html={tex("\\forall a \\in \\mathcal{A}: a = \\text{Tool}_k(\\text{input}_k) \\mapsto \\text{output}_k \\in \\text{AuditLog}", true)} />
</TheoremBlock>

### 2.2 Why Monolithic Architectures Fail

A monolithic LLM approach <InlineEquation html={tex("\\mathcal{F}_{\\text{mono}}(u_t) = \\mathcal{L}(u_t, \\mathcal{D}, s_t)")} /> violates Requirement 1 by construction <CitationRef id={10} />, lacks Requirement 2, and cannot satisfy Requirement 4. Sequential chain architectures <CitationRef id={12} /> address none of these.

## 3. Architecture: LangGraph Agentic System

<TheoremBlock variant="definition" number={2} title="Agentic Workflow System">
An agentic workflow system is a directed graph <InlineEquation html={tex("G = (V, E, \\mathcal{S}, \\delta)")} /> where:

- <InlineEquation html={tex("V = \\{v_{\\text{parse}}, v_{\\text{route}}, v_{\\text{execute}}, v_{\\text{respond}}\\}")} /> are processing nodes
- <InlineEquation html={tex("E \\subseteq V \\times V")} /> are directed edges (conditional transitions)
- <InlineEquation html={tex("\\mathcal{S}")} /> is the shared persistent state space
- <InlineEquation html={tex("\\delta: \\mathcal{S} \\times \\mathcal{U} \\times \\mathcal{A} \\to \\mathcal{S}")} /> is the state transition function

Each node reads from <InlineEquation html={tex("\\mathcal{S}")} />, performs its computation, and writes back <CitationRef id={6} />.
</TheoremBlock>

<TheoremBlock variant="proposition" number={1} title="State Persistence Across Turns">
The workflow state <InlineEquation html={tex("s_t")} /> at turn <InlineEquation html={tex("t")} /> encodes the complete history:

<MathBlock html={tex("s_t = \\delta(s_{t-1}, u_t, a_t) = \\delta(\\ldots\\delta(s_0, u_1, a_1)\\ldots, u_t, a_t)", true)} />
</TheoremBlock>

<TheoremBlock variant="proof">
By induction on <InlineEquation html={tex("t")} />. Base case: <InlineEquation html={tex("s_0")} /> is the initial state. Inductive step: <InlineEquation html={tex("\\delta")} /> is deterministic and <InlineEquation html={tex("\\mathcal{S}")} /> includes all field values, conflict records, and assignment metadata, so no information is lost.
</TheoremBlock>

### 3.1 Why LangGraph Over Vanilla Chains

LangChain's sequential chains implement <InlineEquation html={tex("f = f_n \\circ \\cdots \\circ f_1")} />. LangGraph <CitationRef id={6} /> models the workflow as a directed graph enabling conditional routing:

<MathBlock html={tex("v_{\\text{next}} = \\text{Router}(s_t) = \\begin{cases} v_{\\text{clarify}} & \\text{if ambiguous}(s_t) \\\\ v_{\\text{resolve}} & \\text{if conflict}(s_t) \\\\ v_{\\text{assign}} & \\text{if field\\_pending}(s_t) \\\\ v_{\\text{respond}} & \\text{if complete}(s_t) \\end{cases}", true)} />

### 3.2 MCP Tools as the Action Layer

<TheoremBlock variant="definition" number={3} title="MCP Tool Contract">
A Model Context Protocol tool <InlineEquation html={tex("T_k")} /> is a typed function <CitationRef id={5} />:

<MathBlock html={tex("T_k: \\text{Input}_k \\to \\text{Output}_k", true)} />

The tool set <InlineEquation html={tex("\\mathcal{T} = \\{T_{\\text{template}}, T_{\\text{field}}, T_{\\text{conflict}}, T_{\\text{ambiguity}}\\}")} /> defines the complete action boundary of the agent.
</TheoremBlock>

<TheoremBlock variant="theorem" number={1} title="Action Boundary Enforcement">
<MathBlock html={tex("\\forall a \\in \\mathcal{A},\\; \\exists k: a = T_k(\\text{input}_k) \\implies a \\in \\text{AuditLog}", true)} />
</TheoremBlock>

<TheoremBlock variant="proof">
By construction. The LangGraph execution engine dispatches actions exclusively through <InlineEquation html={tex("\\mathcal{T}")} />. No code path exists outside the tool registry.
</TheoremBlock>

### 3.3 Multi-Layer RAG

<TheoremBlock variant="definition" number={4} title="Layered Retrieval Function">
<MathBlock html={tex("R(q, s_t) = \\bigoplus_{l=1}^{3} R_l(q, s_t, \\mathcal{D}_l)", true)} />

- <InlineEquation html={tex("R_1")} />: Conversation history (session-scoped) <CitationRef id={7} /><CitationRef id={8} />
- <InlineEquation html={tex("R_2")} />: Uploaded documents (user-scoped, format-aware chunking)
- <InlineEquation html={tex("R_3")} />: Institutional knowledge (global scope, prompt template selection)
</TheoremBlock>

Retrieval uses embedding similarity <CitationRef id={9} />:

<MathBlock html={tex("\\text{sim}(q, d_i) = \\frac{\\mathbf{e}_q \\cdot \\mathbf{e}_{d_i}}{\\|\\mathbf{e}_q\\| \\|\\mathbf{e}_{d_i}\\|} \\geq \\tau_l", true)} />

<TheoremBlock variant="proposition" number={2} title="Retrieval Isolation">
For users <InlineEquation html={tex("u_1 \\neq u_2")} />: <InlineEquation html={tex("R_2(q, s_t, \\mathcal{D}_2(u_1)) \\cap R_2(q, s_t, \\mathcal{D}_2(u_2)) = \\emptyset")} />. Enforced by session-scoped embedding indices in PostgreSQL <CitationRef id={14} />.
</TheoremBlock>

### 3.4 PostgreSQL as Unified Backbone

PostgreSQL with pgvector <CitationRef id={14} /> unifies structured metadata and vector embeddings in a single transactional store:

<MathBlock html={tex("\\text{context}(\\text{field}_j, s_t) = \\sigma_{\\text{field}=j}(\\mathcal{S}) \\bowtie \\text{top}_k(R_3(\\text{field}_j, s_t, \\mathcal{D}_3))", true)} />

## 4. Context Isolation Guarantee

<TheoremBlock variant="theorem" number={2} title="Context Isolation">
<MathBlock html={tex("\\mathcal{E}(\\mathcal{L}) \\cap \\mathcal{D}_{\\text{sensitive}} = \\emptyset", true)} />
</TheoremBlock>

<TheoremBlock variant="proof">
The LLM operates only at <InlineEquation html={tex("v_{\\text{parse}}")} /> and <InlineEquation html={tex("v_{\\text{route}}")} />, receiving <InlineEquation html={tex("(u_t, \\mathcal{M})")} />. All data access occurs within MCP tools <InlineEquation html={tex("\\mathcal{T}")} /> in a separate context <CitationRef id={10} />. Tool outputs are post-processed into summaries. Since <InlineEquation html={tex("\\mathcal{M}")} /> contains only schema structure and summaries contain only status, the intersection with <InlineEquation html={tex("\\mathcal{D}_{\\text{sensitive}}")} /> is empty.
</TheoremBlock>

## 5. Impact and Reusability

The system transforms days of drafting into a guided interactive session. The pattern — LangGraph <CitationRef id={6} />, MCP tools <CitationRef id={5} />, multi-layer RAG <CitationRef id={7} />, PostgreSQL — is reusable for any enterprise workflow with institutional knowledge.

That separation is what makes agentic AI viable in regulated environments <CitationRef id={3} /><CitationRef id={13} /> — structural guarantees, not prompt engineering tricks.
