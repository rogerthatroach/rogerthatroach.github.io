## 1. Introduction

Enterprise organizations run on structured processes. Project approvals, budget requests, compliance reviews: these workflows are high-stakes, document-heavy, and riddled with institutional knowledge that lives in people's heads rather than systems.

Most teams throw a chatbot at this. But the user doesn't need a Q&A interface. They need a *collaborator*: something that understands where they are in a multi-step process, holds context across the entire session, pulls the right reference material at the right moment, and resolves contradictions between policy documents without the user knowing there was a conflict.

That's a fundamentally different problem than retrieval-augmented generation <CitationRef id={7} />. It's an orchestration problem over a structured form, inside a regulated-institution governance envelope. This paper formalizes the architectural pattern that solves it, as deployed at PAR Assist inside RBC CFO Group, the first true agentic AI platform approved for production at the bank.

## 2. Problem Formulation

### 2.1 System Requirements

Let <InlineEquation html={tex("\\mathcal{W}")} /> denote the space of enterprise workflows, <InlineEquation html={tex("\\mathcal{U}")} /> the space of user utterances, <InlineEquation html={tex("\\mathcal{D}")} /> the heterogeneous document corpus, <InlineEquation html={tex("\\mathcal{F}_T")} /> the target field set of the chosen template, and <InlineEquation html={tex("\\mathcal{S}")} /> the state space tracking workflow progress. We seek a system <InlineEquation html={tex("\\Phi")} /> that guides users through <InlineEquation html={tex("\\mathcal{W}")} /> while satisfying five formal requirements:

<TheoremBlock variant="definition" number={1} title="System Requirements">
**Requirement 1 (Context Isolation).** The LLM component <InlineEquation html={tex("\\mathcal{L}")} /> must never observe sensitive operational data <InlineEquation html={tex("\\mathcal{D}_{\\text{sensitive}}")} />:

<MathBlock html={tex("\\mathcal{E}(\\mathcal{L}) \\cap \\mathcal{D}_{\\text{sensitive}} = \\emptyset", true)} />

**Requirement 2 (Stateful Orchestration).** The system maintains workflow state <InlineEquation html={tex("s \\in \\mathcal{S}")} /> across an unbounded number of interaction turns:

<MathBlock html={tex("s_{t+1} = \\delta(s_t, u_t, a_t), \\quad \\delta: \\mathcal{S} \\times \\mathcal{U} \\times \\mathcal{A} \\to \\mathcal{S}", true)} />

**Requirement 3 (Field-Group Retrieval).** The corpus <InlineEquation html={tex("\\mathcal{D}")} /> is partitioned into a field-group taxonomy <InlineEquation html={tex("\\mathcal{G} = \\{g_1, \\ldots, g_K\\}")} /> that maps to disjoint target-field subsets <InlineEquation html={tex("\\mathcal{F}_{g_k} \\subseteq \\mathcal{F}_T")} />. Retrieval is two-stage, scoped by relevance at the group level:

<MathBlock html={tex("\\text{retrieve}(q, s_t) = \\bigoplus_{g \\in G(q, s_t)} \\kappa\\bigl(\\operatorname{top}_{k}(q, s_t, \\mathcal{D}_g)\\bigr)", true)} />

where <InlineEquation html={tex("G(q, s_t) \\subseteq \\mathcal{G}")} /> is the relevant-group set from Stage 1, <InlineEquation html={tex("\\operatorname{top}_k")} /> is the per-group Stage 2 similarity search, and <InlineEquation html={tex("\\kappa")} /> is a compression operator fitting the payload to the LLM context budget.

**Requirement 4 (Auditability).** Every system action is a typed, logged tool invocation <CitationRef id={1} /><CitationRef id={2} />:

<MathBlock html={tex("\\forall a \\in \\mathcal{A}: a = T_k(\\text{input}_k) \\mapsto \\text{output}_k \\in \\text{AuditLog}", true)} />

**Requirement 5 (Single-Agent Envelope).** The system must exhibit agentic behaviour (branching dialog, parallel specialized work, multi-turn state) without instantiating multi-agent primitives. Formally, there is exactly one agent-shaped entity <InlineEquation html={tex("\\alpha")} /> in the control surface; all parallelism is delegated to typed tools dispatched by a deterministic orchestrator.
</TheoremBlock>

### 2.2 Why Monolithic Architectures Fail

A monolithic LLM <InlineEquation html={tex("\\Phi_{\\text{mono}}(u_t) = \\mathcal{L}(u_t, \\mathcal{D}, s_t)")} /> violates Requirement 1 by construction <CitationRef id={10} />, lacks Requirement 2, and cannot satisfy Requirement 4. Sequential chain architectures <CitationRef id={12} /> address none of these. A naïve multi-agent design satisfies Requirements 1–4 but violates Requirement 5, which disqualifies it from first-deployment approval in regulated environments.

## 3. Architecture

### 3.1 The Graph

<TheoremBlock variant="definition" number={2} title="Agentic Workflow Graph">
An agentic workflow system is a directed graph <InlineEquation html={tex("G = (V, E, \\mathcal{S}, \\delta)")} /> where:

- <InlineEquation html={tex("V = \\{v_{\\text{intake}}, v_{\\text{template}}, v_{\\text{retrieve}}, v_{\\text{extract}}, v_{\\text{merge}}, v_{\\text{coverage}}, v_{\\text{clarify}}, v_{\\text{respond}}\\}")} /> are processing nodes
- <InlineEquation html={tex("E \\subseteq V \\times V")} /> are directed edges including conditional branches and loops
- <InlineEquation html={tex("\\mathcal{S}")} /> is the shared persistent state space
- <InlineEquation html={tex("\\delta: \\mathcal{S} \\times \\mathcal{U} \\times \\mathcal{A} \\to \\mathcal{S}")} /> is the state transition function

Each node reads from <InlineEquation html={tex("\\mathcal{S}")} />, performs its computation via typed MCP tools <CitationRef id={5} /><CitationRef id={6} />, and writes back.
</TheoremBlock>

Postgres rail carries every write.">
{() => <AgenticArchitecturePAR />}
</VisualizationContainer>

<TheoremBlock variant="proposition" number={1} title="State Persistence Across Turns">
The workflow state <InlineEquation html={tex("s_t")} /> at turn <InlineEquation html={tex("t")} /> encodes the complete interaction history:

<MathBlock html={tex("s_t = \\delta(s_{t-1}, u_t, a_t) = \\delta(\\ldots\\delta(s_0, u_1, a_1)\\ldots, u_t, a_t)", true)} />
</TheoremBlock>

<TheoremBlock variant="proof">
By induction on <InlineEquation html={tex("t")} />. Base case: <InlineEquation html={tex("s_0")} /> is the initial state. Inductive step: <InlineEquation html={tex("\\delta")} /> is deterministic, and <InlineEquation html={tex("\\mathcal{S}")} /> includes the merged extraction dictionary, coverage report, pending follow-ups, and the complete MCP tool-invocation log, so no information is lost. The LangGraph Postgres checkpointer <CitationRef id={6} /> materializes <InlineEquation html={tex("s_t")} /> atomically, making durability a property of the store rather than the application layer.
</TheoremBlock>

### 3.2 Why LangGraph Over Vanilla Chains

LangChain's sequential chains implement <InlineEquation html={tex("f = f_n \\circ \\cdots \\circ f_1")} />, insufficient for conditional-branching workflows. LangGraph <CitationRef id={6} /> models the workflow as a directed graph with conditional routing:

<MathBlock html={tex("v_{\\text{next}} = \\text{Router}(s_t) = \\begin{cases} v_{\\text{clarify}} & \\text{if open\\_followups}(s_t) \\\\ v_{\\text{retrieve}} & \\text{if groups\\_pending}(s_t) \\\\ v_{\\text{extract}} & \\text{if chunks\\_ready}(s_t) \\\\ v_{\\text{merge}} & \\text{if extractions\\_complete}(s_t) \\\\ v_{\\text{respond}} & \\text{if coverage\\_clear}(s_t) \\end{cases}", true)} />

LangGraph was picked for maturity (the most stable production-deployed graph orchestrator at evaluation time) and because PAR drafting is shaped like a graph, not a pipeline. Multi-agent-first frameworks (CrewAI, AutoGen <CitationRef id={4} />) were considered but not deeply evaluated. Their primitives would have violated Requirement 5.

### 3.3 MCP Tools as the Action Layer

<TheoremBlock variant="definition" number={3} title="MCP Tool Contract">
A Model Context Protocol tool <InlineEquation html={tex("T_k")} /> is a typed function <CitationRef id={5} />:

<MathBlock html={tex("T_k: \\text{Input}_k \\to \\text{Output}_k", true)} />

The tool set <InlineEquation html={tex("\\mathcal{T} = \\{T_{\\text{ingest}}, T_{\\text{template}}, T_{\\text{retrieve}_1}, T_{\\text{retrieve}_2}, T_{\\text{compress}}, T_{\\text{extract}}, T_{\\text{merge}}, T_{\\text{coverage}}\\}")} /> defines the complete action boundary of the agent.
</TheoremBlock>

<TheoremBlock variant="theorem" number={1} title="Action Boundary Enforcement">
<MathBlock html={tex("\\forall a \\in \\mathcal{A},\\; \\exists k: a = T_k(\\text{input}_k) \\implies a \\in \\text{AuditLog}", true)} />
</TheoremBlock>

<TheoremBlock variant="proof">
By construction. The LangGraph execution engine dispatches actions exclusively through <InlineEquation html={tex("\\mathcal{T}")} />; no code path exists outside the tool registry. Each invocation writes a typed record to the Postgres audit table as part of the transactional commit of <InlineEquation html={tex("\\delta")} />, so Requirement 4 holds unconditionally.
</TheoremBlock>

Template selection is itself an MCP tool. <InlineEquation html={tex("T_{\\text{template}}")} /> accepts the user's uploaded material and decision-tree answers (project type, spend tier, business line, regulatory exposure) and returns a typed template identifier with confidence and rationale, so even the act of choosing *what PAR we are writing* is auditable.

### 3.4 Two-Stage Field-Group Retrieval

<TheoremBlock variant="definition" number={4} title="Field-Group Taxonomy">
A field-group taxonomy is a partition <InlineEquation html={tex("\\mathcal{G} = \\{g_1, \\ldots, g_K\\}")} /> of the corpus <InlineEquation html={tex("\\mathcal{D} = \\bigsqcup_{g \\in \\mathcal{G}} \\mathcal{D}_g")} /> such that each group maps to a disjoint subset of target fields <InlineEquation html={tex("\\mathcal{F}_g \\subseteq \\mathcal{F}_T")} /> with <InlineEquation html={tex("\\mathcal{F}_{g_i} \\cap \\mathcal{F}_{g_j} = \\emptyset")} /> for <InlineEquation html={tex("i \\neq j")} />. A group aggregates policies, guidelines, historical examples, and few-shot good/bad exemplars for its field subset.
</TheoremBlock>

<TheoremBlock variant="definition" number={5} title="Two-Stage Retrieval Function">
**Stage 1 (group selection).** Given query <InlineEquation html={tex("q")} /> derived from the user's uploaded material and the chosen template, pick the relevant group set:

<MathBlock html={tex("G(q, s_t) = \\{\\, g \\in \\mathcal{G} : \\operatorname{sim}(\\varphi_{\\mathcal{G}}(q, s_t),\\, \\varphi_{\\mathcal{G}}(g)) \\geq \\tau_{\\text{group}} \\,\\}", true)} />

**Stage 2 (chunk retrieval).** For each <InlineEquation html={tex("g \\in G(q, s_t)")} />, run scoped top-<InlineEquation html={tex("k")} /> similarity search over <InlineEquation html={tex("\\mathcal{D}_g")} /> only (production uses <InlineEquation html={tex("k=10")} />):

<MathBlock html={tex("\\mathcal{C}_g = \\operatorname{top}_{k}\\bigl(\\varphi_g(q, s_t),\\, \\mathcal{D}_g\\bigr)", true)} />

**Compression.** Each group's chunk set is reduced by a custom compression operator <InlineEquation html={tex("\\kappa")} /> to fit the per-call LLM context budget:

<MathBlock html={tex("\\hat{\\mathcal{C}}_g = \\kappa(\\mathcal{C}_g) \\quad \\text{where } |\\hat{\\mathcal{C}}_g| \\ll |\\mathcal{C}_g|", true)} />

Similarity uses cosine over an embedding model trained for the corpus <CitationRef id={9} />:

<MathBlock html={tex("\\operatorname{sim}(\\mathbf{e}_q, \\mathbf{e}_{d}) = \\frac{\\mathbf{e}_q \\cdot \\mathbf{e}_{d}}{\\|\\mathbf{e}_q\\|\\,\\|\\mathbf{e}_{d}\\|}", true)} />
</TheoremBlock>

<TheoremBlock variant="proposition" number={2} title="Retrieval Scope Correctness">
Field-group partitioning ensures every retrieved chunk belongs to a group whose target fields are currently extractable:

<MathBlock html={tex("\\forall c \\in \\bigcup_{g \\in G(q, s_t)} \\hat{\\mathcal{C}}_g: \\exists g \\in G(q, s_t) \\text{ such that } c \\in \\mathcal{D}_g \\wedge \\mathcal{F}_g \\cap \\mathcal{F}_T \\neq \\emptyset", true)} />
</TheoremBlock>

<TheoremBlock variant="proof">
Immediate from the partition structure of Definition 4 and the selection criterion in Stage 1. Because <InlineEquation html={tex("\\mathcal{D}_g \\cap \\mathcal{D}_{g'} = \\emptyset")} /> for distinct groups, no chunk appears in unrelated retrieval scopes, which is the structural correction flat top-<InlineEquation html={tex("k")} /> retrieval cannot provide.
</TheoremBlock>

### 3.5 Parallel Extraction Within a Single-Agent Envelope

<TheoremBlock variant="definition" number={6} title="N Parallel Scoped Extraction">
Let <InlineEquation html={tex("N = |G(q, s_t)|")} />. For each relevant group <InlineEquation html={tex("g \\in G(q, s_t)")} />, a scoped extraction call

<MathBlock html={tex("J_g = T_{\\text{extract}}\\bigl(\\mathcal{L},\\,\\hat{\\mathcal{C}}_g,\\, \\mathcal{F}_g,\\, \\Pi_g\\bigr) \\in \\text{JSON}(\\mathcal{F}_g)", true)} />

executes in parallel (at most one LLM invocation per group, bounded by the per-group token budget; production uses Sonnet-4.5 with up to 20 fields per system prompt). <InlineEquation html={tex("\\Pi_g")} /> carries per-group guardrails, instructions, and few-shot good/bad exemplars. Each output <InlineEquation html={tex("J_g")} /> is a JSON dictionary whose keys are the target fields <InlineEquation html={tex("\\mathcal{F}_g")} />.
</TheoremBlock>

<TheoremBlock variant="definition" number={7} title="Dictionary-Union Merge">
<MathBlock html={tex("J^* = \\bigcup_{g \\in G(q, s_t)} J_g", true)} />

Because <InlineEquation html={tex("\\mathcal{F}_{g_i} \\cap \\mathcal{F}_{g_j} = \\emptyset")} /> for <InlineEquation html={tex("i \\neq j")} /> (Definition 4), the union is well-defined without synthesis: at most one extraction call contributes a value per field.
</TheoremBlock>

<TheoremBlock variant="theorem" number={2} title="Single-Agent Envelope">
Under the construction above, parallelism <InlineEquation html={tex("N > 1")} /> does not imply multiple agents. The deterministic orchestrator <InlineEquation html={tex("G")} /> dispatches <InlineEquation html={tex("N")} /> instances of <InlineEquation html={tex("T_{\\text{extract}}")} /> (each a typed tool invocation, not an autonomous agent) and merges their outputs via Definition 7. Exactly one agent-shaped entity <InlineEquation html={tex("\\alpha")} /> exists in the control surface; Requirement 5 holds.
</TheoremBlock>

<TheoremBlock variant="proof">
A multi-agent system would require (i) agents with independent control, (ii) inter-agent messaging as a first-class primitive, and (iii) emergent coordination. Our construction has none: control is vested in the LangGraph router, "messaging" is replaced by deterministic dict-union merge over disjoint field domains (Definition 7), and "coordination" reduces to graph topology. The same <InlineEquation html={tex("N")} /> that drives parallelism in a multi-agent shape is absorbed into the *tool-dispatch fan-out* in our shape, which is exactly the transformation required to fit the single-agent envelope.
</TheoremBlock>

### 3.6 Coverage and Follow-Up Loop

<TheoremBlock variant="definition" number={8} title="Coverage Operator">
<MathBlock html={tex("\\langle\\, \\rho_t,\\, \\mathcal{Q}_t \\,\\rangle = T_{\\text{coverage}}(J^*,\\, \\mathcal{F}_T,\\, \\text{guidelines},\\, \\text{exemplars})", true)} />

where <InlineEquation html={tex("\\rho_t \\in [0,1]")} /> is the coverage ratio (fraction of required fields populated with defensible values) and <InlineEquation html={tex("\\mathcal{Q}_t")} /> is the set of intelligent follow-up questions. The graph transitions to <InlineEquation html={tex("v_{\\text{respond}}")} /> iff <InlineEquation html={tex("\\rho_t \\geq \\tau_{\\text{ship}} \\wedge \\mathcal{Q}_t = \\emptyset")} />; otherwise the session loops through <InlineEquation html={tex("v_{\\text{clarify}}")} />.
</TheoremBlock>

### 3.7 PostgreSQL as the Unified Store

LangGraph checkpoints, tool-invocation logs, raw ingested content, mapped intermediates, pgvector embeddings, and the audit trail all reside in one Postgres instance <CitationRef id={14} />:

<MathBlock html={tex("\\text{store}(s_t) = \\underbrace{\\text{ckpt}(s_t)}_{\\text{LangGraph}} \\cup \\underbrace{\\text{log}(a_{0:t})}_{\\text{MCP calls}} \\cup \\underbrace{\\mathcal{D}_{\\text{raw}} \\cup \\mathcal{D}_{\\text{mapped}}}_{\\text{content}} \\cup \\underbrace{\\{\\mathbf{e}_d\\}_{d \\in \\mathcal{D}}}_{\\text{pgvector}}", true)} />

One transactional store means a session's complete provenance is reconstructable with a handful of joins, and the single atomic snapshot of <InlineEquation html={tex("\\delta(s_t, u_t, a_t)")} /> avoids cross-store consistency windows.

## 4. Context Isolation Guarantee

<TheoremBlock variant="theorem" number={3} title="Context Isolation">
<MathBlock html={tex("\\mathcal{E}(\\mathcal{L}) \\cap \\mathcal{D}_{\\text{sensitive}} = \\emptyset", true)} />
</TheoremBlock>

<TheoremBlock variant="proof">
The LLM operates at the intake/template node and at the <InlineEquation html={tex("N")} /> extraction nodes, receiving <InlineEquation html={tex("(u_t, \\hat{\\mathcal{C}}_g, \\mathcal{F}_g, \\Pi_g)")} /> in each call. Compressed chunks <InlineEquation html={tex("\\hat{\\mathcal{C}}_g")} /> are drawn exclusively from the public policy/guideline corpus <InlineEquation html={tex("\\mathcal{D}_g \\subseteq \\mathcal{D} \\setminus \\mathcal{D}_{\\text{sensitive}}")} />; sensitive operational data flows only through MCP tools <InlineEquation html={tex("\\mathcal{T}")} /> in a separate context <CitationRef id={10} />. Tool outputs rejoining the agent context are post-processed into status summaries containing no raw sensitive payload. The intersection with <InlineEquation html={tex("\\mathcal{D}_{\\text{sensitive}}")} /> is empty by construction.
</TheoremBlock>

## 5. Impact and Reusability

PAR Assist transformed multi-week governance drafting into a guided interactive session, with every extracted field cited back to a policy, guideline, or few-shot example. Pilot launched April 2026; enterprise rollout across RBC business lines through Q2/Q3 2026.

The five-piece pattern (LangGraph <CitationRef id={6} />, MCP tool boundary <CitationRef id={5} />, two-stage field-group retrieval <CitationRef id={7} />, N parallel scoped extraction with dict-union merge, unified Postgres backbone) is reusable for any enterprise workflow with (i) a structured target form, (ii) institutional knowledge across heterogeneous sources, and (iii) regulatory audit requirements. The v2 evolution replaces the single-agent envelope with a multi-agent *skills* framework composed over the same MCP tool registry; v1 is the substrate v2 extends.

<TheoremBlock variant="remark">
<MathBlock html={tex("\\underbrace{\\mathcal{L}}_{\\text{LLMs reason}} \\circ \\underbrace{\\mathcal{T}}_{\\text{tools act}} \\circ \\underbrace{\\mathcal{R}}_{\\text{code controls flow}} \\circ \\underbrace{\\mathcal{S}}_{\\text{Postgres remembers}}", true)} />

That separation is what makes agentic AI viable in regulated environments <CitationRef id={3} /><CitationRef id={13} />: structural guarantees, not prompt engineering tricks, and a single-agent envelope that trades maximum capability today for an approved substrate that v2 skills already sit on top of.
</TheoremBlock>

Working code, synthetic data, and reproducible plots: [`rogerthatroach/prometheus-multi-agent-retrieval`](https://github.com/rogerthatroach/prometheus-multi-agent-retrieval).

The two-stage retrieval + parallel extraction + coverage-loop architecture this post describes runs end-to-end on 50 synthetic PAR docs.
