Finance leadership asks questions that cross three domains in one sentence: *"headcount by division, crossed with open positions, crossed with tenure."* The answer needs to be deterministic under audit, scoped by entitlement at every level of the organization, and returned in milliseconds — not days.

There is one pattern the industry reaches for here: give the LLM full access and let it figure it out. One model, function-calling into the financial data store, iterating with a ReAct-style loop until it lands on an answer. Fast to prototype. Easy to demo. Impossible to ship inside regulated finance.

Astraeus went the other way. Up to six GPT-4.1 calls per query, zero of them touching operational data. A deterministic Cython-compiled Python layer between two dashed walls. A permission-to-SQL entitlement cascade applied *before* compute, not after. This post unpacks the four architectural decisions that separate LLM-as-Router from the seductive pattern — with the alternatives that were seriously considered, the constraints each one answered, and the reasoning that produced the call. The narrative of making those calls — under pressure, across stakeholders — lives in the [builder companion](/blog/astraeus-llm-as-router). The formal proofs of data-confidentiality and entitlement safety live in the [formal post](/blog/agentic-ai).

## The four architectural decisions

Four calls separate LLM-as-Router from the monolithic pattern. Each one mapped to an explicit options comparison before it landed.

<DecisionRationale option="LLM-as-Router + deterministic compute">
Three reasons, in order. First — **confidentiality by construction**. Every alternative where the LLM reads operational data leaves confidentiality as a prompt-engineering and vendor-contract discipline. LLM-as-Router makes it a type-system property: the model's context window never receives a row of employee data, because the code path that would feed it one does not exist. That is the only formulation that clears regulatory review without a footnote.

Second — **determinism as a pre-condition for use.** Finance cannot accept two different numbers for the same question. The compute layer has to be deterministic. Once you accept that, the question becomes: do I want the LLM upstream of determinism (it routes requests into deterministic compute) or downstream (it reads the deterministic output to *interpret* it)? Upstream is safer for audit: routing errors are visible and recoverable; interpretation errors contaminate the answer.

Third — **entitlement becomes enforceable before compute runs.** Because deterministic code owns the data path, the permission-to-SQL entitlement cascade gets to run *before* the compute step, not woven into an LLM prompt. The audit story is short: *here is the user, here is the cube-permission lookup, here is the security group, here is the employee set, here are the events they can see, here is the SQL WHERE clause.* Every step is a function call, logged, typed, replayable.

The cost of this shape is that the deterministic compute layer has to be built and maintained. That cost was obviously the right trade, but it was the part that took real engineering.
</DecisionRationale>

### Decision 2 · Computation layer

Once the LLM is out of the compute path, *what* does compute? Event-level ins-outs math across ~40K leaf-level events with factorial cross-combinations — every candidate answer needs netting semantics applied over events, not aggregates over snapshots. The compute layer is the place where a millisecond answer lives or dies.

<DecisionRationale option="Cython-compiled Python">
Two reasons. First — **match the team shape.** Python was already the language for everything else in the stack: routing logic, ORM, serialization, audit-log writes. Cython gave the compute layer C-level speed for the hot paths without forking the codebase into two languages. The reviewer pool stayed the same. The hiring pipeline stayed the same. The on-call runbook stayed the same.

Second — **profile first, then compile.** Cython's annotation mode shows which Python lines are still Python-level (with a visible interpreter cost) and which have been lowered to C. That made optimization a surgical activity — find the lines that matter, annotate with type declarations, recompile, re-measure. The system only had to meet millisecond latency on the hot paths; the rest could stay readable Python. Pure-Rust or pure-SQL alternatives would have forced an all-or-nothing rewrite.

The SQL-only path was genuinely considered and ruled out during implementation — event-level netting across factorial cross-domain combinations produces query plans that the Postgres planner handles inconsistently at this scale. The computation is more readable as imperative Python over a well-indexed event log than as a multi-hundred-line window-function SQL query.
</DecisionRationale>

Who is allowed to see which slice? The answer was already encoded, across years, in EPM: domain permissions → access groups → entities → events → SQL. The question was *where* to apply it.

<DecisionRationale option="EPM passthrough, pre-compute">
The deciding property was **reuse of the source of truth**. The bank had spent years curating the EPM catalog — domain permissions, security groups, employee assignments, events — with a review and change-management process that already satisfied auditors. Any entitlement model that did not reuse that catalog forked the source of truth and introduced a drift risk that regulated finance will not accept.

Pre-compute placement came from a narrower argument: *least privilege.* A query result that was post-filtered from a larger computation means the computation briefly ran over data the user was not entitled to — and the audit story has to either explain that away or prove that no information leaked through the differential. Pre-compute filtering makes the audit trivial: the computation input itself is the entitled set.

The five-step cascade (domain → access → entities → events → SQL) was the real engineering. Each step is a function call with typed inputs and outputs; each is tested; each writes to the Postgres audit log. Any question of the form "could user X ever have seen value Y?" reduces to replaying the cascade.
</DecisionRationale>

### Decision 4 · Synthesis strategy

Aggregates cross back up from the deterministic layer. The LLM picks them up for the final step: phrasing the answer, combining cross-domain results, generating the right output format. The question is *how many* LLM calls participate in that final step.

The synthesis workload is bimodal. Most queries are single-domain and respond cleanly to a single synthesis call — a user asking "what is headcount for the Technology division this quarter" does not benefit from three parallel agents. The phase-1 router already classifies intent to decide whether the query is in scope at all. Extending that classification to pick the synthesis path is cheap: the information the router needs (how many domains the query touches, whether cross-domain reconciliation is needed) is already on its plate. That makes the hybrid architecturally clean rather than being bolted on.

The deterministic combine step after path B is the audit anchor. When three sub-agents produce cross-domain answers, the reconciliation — how the rollups interact, which base number anchors the cross-domain join — is not a fourth LLM call. It's deterministic code with a typed aggregate-in / aggregate-out signature. The final user-facing phrasing is LLM; the numerical reconciliation is not.

## The permission cascade in detail

The five-step cascade is the engineering that made this architecture shippable inside governance. Each step is a deterministic function with typed inputs and outputs, logged to Postgres, replayable from the audit log.

1. **Domain permissions.** The EPM organizes data into domains; the user identity resolves to a set of domain permissions. This is a direct lookup against the entitlement catalog.
2. **Access groups.** Domain permissions map to access groups with finer-grained scope — business line, geography, grain. The mapping is maintained in EPM and pulled via a periodic catalog sync.
3. **Entities.** Access groups map to the employee records the user is entitled to see — visible headcount, visible organizational rollups.
4. **Events.** The atomic routing unit (the leaf-level routing record). Employee assignments resolve to the ~40K-event subset — the leaf-level units that event-level math operates on.
5. **SQL WHERE clause.** The event set materializes into a SQL predicate that Postgres enforces at query time. Every compute query carries this predicate; no path exists that computes over a larger scope and filters later.

The decision to reuse EPM rather than build a parallel model is what let this ship. The bank had years of governance process around the EPM catalog — review, change management, auditor sign-off. Forking the permission model outside EPM would have triggered a separate governance review on the new system, and the new system would have drifted from the old one the moment EPM changed.

## Why 40K is a factorial number

The scale callout in the architecture diagram reads *"40K! factorial combinations."* The exclamation mark is not emphasis — it is the factorial operator. Any cross-domain query can, in the limit, slice the ~40K leaf-level event set in any permutation consistent with the entitlement filter. Previously the finance organization could not ask those questions interactively at all; the analyst time to produce one such slice was measured in days of spreadsheet work.

Cython's role in this is blunt: the event-level netting math inside the hot loop runs in C. The rest of the layer — IO, entitlement resolution, serialization — stays in idiomatic Python. Profiling showed the hot loop was responsible for the majority of runtime on cross-domain queries, so that's where the Cython annotations concentrate.

The *previously impossible* framing is not marketing. Before Astraeus, finance-leadership questions of the form "how does headcount cross with open positions by division, at Q2" either got answered with a multi-day analyst turnaround or did not get answered at all. The combinatorial size of the question space made interactive slicing infeasible. LLM-as-Router plus Cython-compiled compute turns the question space into an interactive one.

## Coverage, guardrails, and how the walls actually hold

Three properties the architecture has to maintain under real queries — including the ones that fail.

**What happens when the parse fails.** The LLM 1 call produces a structured intent object or an error. If the intent object is malformed — no domain class resolvable, no output shape — the system returns a clarification prompt rather than guessing. The route call (LLM 2) does not fire on a failed parse. No data access occurs. The failure mode is a visible refusal, not a silent wrong answer.

**What happens when the route says out-of-scope.** The route call explicitly decides whether the query is inside Astraeus domain. Questions about payroll processing, HR performance reviews, benefits administration — all out of scope. On an out-of-scope route, execution terminates at the wall. Neither the deterministic compute layer nor the downstream synthesis runs. The user sees a scoped answer ("Astraeus covers headcount, hiring, and cost — for payroll detail, see [other system]").

**How the walls actually prevent leakage.** Wall 1 (between LLM intent and deterministic compute) is enforced by code structure: the functions that call the LLM do not have an argument that could be operational data. The functions that call compute do not have an argument that could be an LLM. That separation is a property of the type signatures, not a convention. Wall 2 (between deterministic compute and LLM synthesis) is enforced the same way: synthesis functions receive aggregates, not events. A reviewer can audit the wall by looking at function signatures.

The single Postgres backbone underneath is the audit anchor — every LLM call, every compute step, every entitlement resolution writes a structured record. Replay of any answer reduces to replaying the log.

## Timeline + velocity

Astraeus went from the first architectural sketch to production in ~5 months, with v2 cycles continuing after. The architecture stabilized early; most of the engineering time after month two went into the permission cascade, the Cython-compiled compute layer, and the event-level ins-outs math. Stakeholder conversations — explaining, repeatedly, why LLM-as-Router was the right shape rather than the seductive monolithic pattern — ran in parallel through the first three months. The [builder companion](/blog/astraeus-llm-as-router) has the narrative of that stretch.

## What's next — Astraeus v2

The next evolution extends the pattern without changing its shape. Three directions on the table:

- **Skill composition.** Path B sub-agents become named skills with declared input/output contracts. A new domain is added by registering a skill, not by forking the synthesis code.
- **Richer routing.** The phase-1 router gains access to a small set of deterministic classifiers (question-shape, temporal-shape, cross-domain-shape) that inform path selection more precisely.
- **Expanded scope.** Additional domains come online behind the same pattern — forecasting, variance analysis — each as a new deterministic compute module plus a new sub-agent for synthesis. The envelope does not change.

The constraints that shaped v1 still hold. Every v2 capability still has to pass the same four tests: deterministic under audit, entitled by construction, millisecond at scale, reusing the source of truth.

## What transfers

The LLM-as-Router pattern is not specific to CFO analytics. It applies anywhere:

- The output has to be deterministic under audit
- Sensitive data must not enter the model context window
- Permission enforcement must be provable, not probabilistic
- The cost of a wrong answer exceeds the cost of a slightly less elegant architecture

Astraeus was the first production shape this pattern took inside RBC; it now underlies other systems — Prometheus's agentic envelope is the same LLM-as-Router principle with a richer action vocabulary (see [the PAR practitioner post](/blog/enterprise-agentic-ai-framework) for the MCP-tools variant). Aegis v2's text-to-SQL is LLM-as-Router with tighter intent classification and a guardrailed SQL surface.

The pattern scales across system types because the constraints are shared. The seductive option — give the LLM full access, let it figure it out — fails the same four tests in every regulated-enterprise setting. LLM-as-Router is the compact answer to those tests.
