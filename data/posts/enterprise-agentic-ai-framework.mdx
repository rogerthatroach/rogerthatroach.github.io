Enterprise organizations run on structured processes. Project approvals, budget requests, compliance reviews — high-stakes, document-heavy workflows riddled with institutional knowledge that lives in people's heads rather than systems.

Most teams throw a chatbot at this. But the user doesn't need a Q&A interface. They need a **collaborator** — something that understands where they are in a multi-step process, holds context across the entire session, pulls the right reference material at the right moment, and resolves contradictions between policy documents without the user knowing there was a conflict.

That's not retrieval-augmented generation <CitationRef id={7} />. That's orchestration. This post walks through the architecture pattern that solves it, at <Glossed>PAR Assist</Glossed> inside <Glossed>RBC CFO Group</Glossed>.

## Constraints before choices

Four constraints shaped every architectural decision below. Two are hard (regulatory / technical absolutes), one soft (institutional momentum), one regulatory (auditability). Every downstream choice satisfies these first; aesthetics second.

## The three architectural decisions

Three design calls separate a PAR-Assist-shaped system from a retrieval chatbot. Each one mapped to an explicit options comparison before it landed.

### Decision 1 · Orchestration framework

<DecisionRationale option="LangGraph">
PAR workflows branch conditionally. A user picks a template, the agent extracts fields, sometimes a field is ambiguous (loop to clarify), sometimes two policies conflict (loop to resolve), sometimes the project type changes mid-draft (loop back to template selection). That shape is a graph, not a pipeline.

The second reason: LangGraph's persistent-state model maps directly to Constraint 2. State isn't a memory add-on — it's the thing the graph operates over. Every node reads from and writes to the shared state, so conversation history, field values, and conflict records live in one place.

Sequential chains would have forced us to bolt on both routing and memory. Hand-rolled FSM would have forced us to rebuild LangGraph's tracing surface. LangGraph was the shortest path to the constraints.
</DecisionRationale>

### Decision 2 · Action boundary

Every action the agent takes — select a template, extract a field, resolve a conflict — has to pass through a typed, logged contract. That's how Constraint 3 (auditability) becomes a property of the system, not a promise.

<DecisionRationale option="MCP tools">
The property we care about is: *"every action the agent takes is in the audit log."* With MCP, that's true by construction — the engine dispatches actions exclusively through the tool registry, so no code path exists outside it. With free-form function calling, the same property is a prompt-engineering discipline: it could be true, but it's not structurally guaranteed.

Regulated finance cannot accept "probably true." MCP made the guarantee structural, not aspirational.
</DecisionRationale>

### Decision 3 · Retrieval architecture

Three very different shapes of content feed the agent: the ongoing conversation, documents the user uploads mid-session, and the institutional knowledge base (policies, templates, historical examples). Treating all three with one retrieval function would have been cheaper to build but would have blurred the boundaries between them.

<OptionsConsidered
  decision="Single retrieval index, or layered retrievers per source?"
  options={[

        'Retrieval orchestration logic is non-trivial — results have to be combined in a way that respects relative thresholds',
      ],

    },
    {
      name: 'Graph RAG',
      summary: '[PLACEHOLDER — was this evaluated given policy-document citation structure? See brief §7.]',

<DecisionRationale option="Multi-layer RAG">
Session-scoped isolation is the killer argument. When a user uploads a PDF mid-session — a policy memo, a template, a draft — that content should never appear in another user's retrieval results. With three separate layers, that property is structural: user-uploaded content lives in a session-scoped index and gets torn down with the session. With a single index, session isolation would be a post-retrieval filter, which is a correctness-as-runtime-check pattern — acceptable in some contexts, not in regulated finance.

The format-aware chunking is the second reason. Policies live in sections with citation back-references. Transcripts are line-oriented. PDFs have headings. One chunker for all of them would underserve all of them.
</DecisionRationale>

## What transfers

The pattern — **conditional-graph orchestration + typed action boundary + layered retrieval + unified state store** — transfers to any enterprise workflow that combines (a) multi-turn state, (b) heterogeneous sources, and (c) regulatory audit requirements. PAR drafting is one instance. Budget-request review, compliance audit, credit-memo drafting, M&A checklist work — same skeleton, different domain anchors.

The substrate changes; the shape doesn't. That's what makes agentic AI viable in regulated environments: structural guarantees from the architecture, not prompt-engineering discipline at the margin.

Let <InlineEquation html={tex("\\mathcal{W}")} /> denote the space of enterprise workflows, <InlineEquation html={tex("\\mathcal{U}")} /> user utterances, <InlineEquation html={tex("\\mathcal{D}")} /> the document corpus, and <InlineEquation html={tex("\\mathcal{S}")} /> the workflow-state space. System <InlineEquation html={tex("\\mathcal{F}")} /> satisfies:

<MathBlock html={tex("s_{t+1} = \\delta(s_t, u_t, a_t), \\quad \\delta: \\mathcal{S} \\times \\mathcal{U} \\times \\mathcal{A} \\to \\mathcal{S}", true)} />

<TheoremBlock variant="definition" number={2} title="Agentic workflow graph">
An agentic workflow system is a directed graph <InlineEquation html={tex("G = (V, E, \\mathcal{S}, \\delta)")} /> where <InlineEquation html={tex("V = \\{v_{\\text{parse}}, v_{\\text{route}}, v_{\\text{execute}}, v_{\\text{respond}}\\}")} />, <InlineEquation html={tex("E \\subseteq V \\times V")} />, and each node reads from and writes to <InlineEquation html={tex("\\mathcal{S}")} /><CitationRef id={6} />.
</TheoremBlock>

<MathBlock html={tex("v_{\\text{next}} = \\text{Router}(s_t) = \\begin{cases} v_{\\text{clarify}} & \\text{if ambiguous}(s_t) \\\\ v_{\\text{resolve}} & \\text{if conflict}(s_t) \\\\ v_{\\text{assign}} & \\text{if field\\_pending}(s_t) \\\\ v_{\\text{respond}} & \\text{if complete}(s_t) \\end{cases}", true)} />

<TheoremBlock variant="theorem" number={1} title="Context isolation">
<MathBlock html={tex("\\mathcal{E}(\\mathcal{L}) \\cap \\mathcal{D}_{\\text{sensitive}} = \\emptyset", true)} />

By construction: the LLM operates only at <InlineEquation html={tex("v_{\\text{parse}}")} /> and <InlineEquation html={tex("v_{\\text{route}}")} />, receiving <InlineEquation html={tex("(u_t, \\mathcal{M})")} />. All data access occurs within MCP tools in a separate context<CitationRef id={10} />. Tool outputs are post-processed into summaries before rejoining the agent context. Since <InlineEquation html={tex("\\mathcal{M}")} /> contains only schema structure and summaries contain only status, the intersection with <InlineEquation html={tex("\\mathcal{D}_{\\text{sensitive}}")} /> is empty.
</TheoremBlock>

<MathBlock html={tex("\\underbrace{\\mathcal{L}}_{\\text{LLMs reason}} \\circ \\underbrace{\\mathcal{T}}_{\\text{tools act}} \\circ \\underbrace{\\mathcal{R}}_{\\text{code controls flow}} \\circ \\underbrace{\\mathcal{S}}_{\\text{database remembers}}", true)} />

That separation is what makes agentic AI viable in regulated environments<CitationRef id={3} /><CitationRef id={13} /> — structural guarantees, not prompt engineering tricks.
