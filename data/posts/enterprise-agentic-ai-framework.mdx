Enterprise organizations run on structured processes. Project approvals, budget requests, compliance reviews: high-stakes, document-heavy workflows riddled with institutional knowledge that lives in people's heads rather than systems.

Most teams throw a chatbot at this. But the user doesn't need a Q&A interface. They need a **collaborator**: something that understands where they are in a multi-step process, holds context across the session, pulls the right reference material at the right moment, and resolves contradictions between policy documents without the user knowing there was a conflict.

That's not retrieval-augmented generation <CitationRef id={7} />. That's orchestration over a structured form. This post walks through the architecture at <Glossed>PAR Assist</Glossed> inside <Glossed>RBC CFO Group</Glossed>, the first true agentic system approved for production deployment at the bank.

## The four architectural decisions

Four calls separate PAR Assist from a clever chatbot. Each one mapped to an explicit options comparison before it landed.

Two reasons, in order. First: LangGraph was the most mature, stable orchestration library at the time the call was made (stable release, production deployments at other shops, active maintenance). In a regulated-finance first-deployment context, *stable and maintained* outranks *elegant on paper*.

Second: the drafting session is shaped like a graph. A user picks a template, the agent retrieves relevant field groups, extracts candidate values, sometimes a field is ambiguous (loop to clarify), sometimes two guideline clauses conflict (loop to resolve), sometimes the project classification changes mid-session (loop back to template selection). That topology is a graph, not a pipeline, and LangGraph's checkpointer wires directly into Postgres, which gave us session durability without building a second persistence layer.
</DecisionRationale>

### Decision 2 · Action boundary

Every action the agent takes (pick a template, retrieve a group, extract fields, run a coverage check) has to pass through a typed, logged contract. That's how auditability becomes a property of the system rather than a promise.

<DecisionRationale option="MCP tools">
The property we care about is: *every action the agent takes lands in the audit log.* With MCP, that is true by construction: the engine dispatches actions exclusively through the tool registry, so no code path exists outside it. With free-form function calling, the same property is a prompt-engineering discipline: it could be true, but it is not structurally guaranteed.

Regulated finance cannot accept "probably true." MCP made the guarantee structural.

A concrete example: template selection is itself an MCP tool. It takes the user's uploaded material and their responses to decision-tree questions (project type, spend tier, business line, regulatory exposure) and returns a typed template-id with a confidence score. The tool is how the agent "decides" which PAR template fits, and every call into it is logged, inputs and outputs both, so the selection is auditable end-to-end.
</DecisionRationale>

### Decision 3 · Retrieval shape

PAR templates are large. Across the bank's PAR types there are hundreds of fields, and most of them repeat, partially, across templates. Flat chunk retrieval against policy and guideline documents would produce noisy top-k for any given drafting session: too much relevant-looking text, not enough that actually maps to a specific field.

The shape we needed is **field-group retrieval**: organize the knowledge base by the *target* the LLM is extracting toward, not by source document.

<DecisionRationale option="Two-stage retrieval with group taxonomy">
The insight is that a PAR has a known target shape. We know which fields we're trying to fill before we retrieve anything. If we organize knowledge by those fields (group related fields together across templates, attach the relevant policy/guideline/example chunks to each group) retrieval becomes a lookup into a structure the LLM already understands.

**Stage 1** picks relevant groups for the current session (by uploaded-doc similarity + template-selection context). **Stage 2** runs similarity search within each group, returning top-10 chunks per group. A custom compression pass (details of the compression will be covered in a follow-up post) reduces those ten chunks into a dense, signal-heavy representation the LLM can fit alongside up to 20 fields' worth of rich metadata in a single Sonnet-4.5 system prompt.

Flat RAG would work for a Q&A chatbot. It does not work for structured field extraction across a template where fields are categorically different.
</DecisionRationale>

### Decision 4 · Extraction within the single-agent envelope

This is where the governance constraint (single-agent envelope) met the problem shape (parallel work across field groups). The natural answer would have been multi-agent orchestration: one sub-agent per group, each a specialist. Governance said no.

<DecisionRationale option="Parallel LLM calls, one per relevant group">
The relevant groups picked by stage-1 retrieval become the parallelism. Up to N calls fire at once, each one carrying its group's compressed chunks, up to 20 fields of metadata for that group, and the guardrails / few-shot good-and-bad examples specific to its scope. Each returns a JSON dictionary of extracted field values.

The merge step is a dictionary union. Because groups are disjoint field sets by construction, conflicts are rare and localized; when they happen, last-writer-wins with a flag raised for the coverage analyzer downstream (leaning toward union rather than a synthesis pass; revisitable if we find conflicts becoming routine).

The result: specialized, scoped extraction that runs in parallel and merges deterministically, all inside a single-agent envelope. The LLM never reasons about "I am coordinating sub-agents"; the graph does that. What it sees is *tools that happen to return structured data.*
</DecisionRationale>

## The Postgres backbone

One of the less-glamorous decisions that made a lot of the rest of the system cheap: **everything goes in Postgres**. LangGraph checkpoints, tool-invocation logs, the merged session state, raw uploaded content, mapped/normalized intermediates, the embeddings used for retrieval, the audit trail. One store.

That's not the default for a system like this: retrieval usually sits in a dedicated vector store, logs in a time-series database, state in Redis or similar. One-store is unusual.

The reasons are pragmatic. pgvector is good enough for the retrieval scale PAR Assist operates at. Having state, logs, and embeddings in one transactional store means a single session's entire provenance is reconstructable with a handful of joins, which matters for audit. Backup, replication, and access control only need to be solved once. And when the graph's Postgres-backed checkpointer takes a snapshot, it captures the whole session atomically, no cross-store consistency window to worry about.

Two cons, honestly stated: pgvector is not the fastest vector store at the high end, and a single Postgres instance is a single blast radius. Both are acceptable at current scale; both are revisitable if scale changes.

## Coverage, guardrails, and few-shot examples

Three quality layers run through the system, each at a different node in the graph:

1. **Per-group guardrails:** applied inside each parallel extraction call. The group's system prompt carries the constraints: what the field *must* be, what the field *cannot* be, what the field *is allowed to be phrased as.* The LLM can't emit a value that fails a guardrail because the guardrail is in the prompt, enforced by the few-shot bad examples.

2. **Per-group few-shot good/bad examples:** small, curated libraries of real-world extracted values and why they're good or bad. These do more work than abstract rules because they show the LLM what "defensible" actually looks like for a given field.

3. **Session-level coverage analyzer:** runs after the merge. Checks the full merged state against the guideline set. Produces a coverage report + a prioritized list of intelligent follow-ups. The follow-ups are what turn PAR Assist from a field extractor into a collaborator: "You haven't addressed regulatory exposure for this spend tier. Here are three questions."

## War story: living inside a single-agent envelope

The problem wanted multiple agents. A PAR has disjoint field groups (risk, financial structure, regulatory impact, implementation plan) that look like separate specialist domains. The natural architecture would have been one sub-agent per domain, each a specialist with its own tools and guardrails, coordinated by an orchestrator.

Governance said no. The first agentic framework approved for deployment at the bank was a single-agent envelope: one agent, one approved scope, no multi-agent orchestration in v1. The right engineering answer was structurally unavailable.

What we built instead was the multi-agent *behavior* without the multi-agent *shape*. The parallel extraction calls are not sub-agents; they are tool invocations. The orchestration is deterministic: the graph decides which groups are in play, the graph fires the N calls, the graph merges the results. The LLM never reasons about "I am coordinating specialist agents." It sees a sequence of extraction tools, each returning structured data.

From the outside, it looks agentic: branching dialogs, multi-step reasoning, parallel specialized work. From the governance review, it is one agent calling tools. Both things are true. The architecture sits exactly on that seam.

The cost was design discipline. Every time we reached for a multi-agent pattern (persistent cross-agent memory, inter-agent negotiation, agent-on-agent reasoning) we had to ask whether that could be expressed as deterministic graph logic plus MCP tools, or whether it fundamentally required multi-agent primitives. In v1, everything that needed the latter got deferred to v2.

**The lesson that generalizes:** in regulated environments, the first agentic deployment is not about maximum capability. It is about building the envelope in a way that the next envelope (larger) inherits without rewrite. PAR Assist v1 is single-agent. The groups, the parallel calls, the per-group guardrails, the coverage analyzer, the Postgres backbone, the MCP tool surface: all of these become the substrate v2 runs on. The LangGraph graph's next evolution is multi-agent *skills* composed over the same tool registry. We are not building v1 as a throwaway; we are building v1 as the part of v2 that already passed review.

## What's next: skills

v2 of the framework moves from single-agent to a LangGraph multi-agent composition where the primitive is a **skill** rather than a tool. A skill is a bounded capability: either a small focused LLM call (shorter prompt, specialized output), an added-context step (injects template-specific or role-specific context before a call), or a deterministic algorithmic step (rule-based transformation, validation, or lookup).

Skills are composable. A workflow is a graph over skills. A "draft-this-PAR" is no longer one agent with tools; it is a coordination of skills, each with its own approved scope, its own audit surface, its own testable contract. The single-agent envelope becomes one specialization inside the skill catalog.

Crucially: because PAR Assist v1 built everything on MCP tools, the tool registry is already the substrate v2's skills compose over. No rewrite. The v1 architecture is the subgraph v2 extends.

## What transfers

The pattern (**governance-shaped orchestration + typed action boundary + field-group retrieval + parallel scoped extraction + unified state store**) transfers to any enterprise workflow that combines (a) multi-turn state, (b) structured-form target outputs, and (c) regulatory audit requirements. PAR drafting is one instance. Budget-request review, compliance audit, credit-memo drafting, M&A checklist work: same skeleton, different domain anchors.

The substrate changes; the shape does not. That is what makes agentic AI viable in regulated environments: structural guarantees from the architecture, not prompt-engineering discipline at the margin.
